## Trampas conocidas

1. **Workers=1 es obligatorio.** ChromaDB usa SQLite. Aumentar workers rompe.
2. **Orden ingest ‚Üí hot-reload.** `ingest.py` primero, luego `POST /admin/refresh`.
3. **`data/**/*.md` en .gitignore.** No committees informacion personal.
4. **LiteLLM es el proxy.** Las API keys reales de Groq NO estan en este repo.
5. **`print()` prohibido.** Usar `logging` con niveles.
6. **`entity_name` sin `.md`** en frontmatter. Archivos fisicos si tienen extension.
7. **`llama-8b` alucina con listas.** El cambio a 70B (T-027) deberia resolverlo.
8. **`_check_confidence` es debil.** Solo busca entity_name en la respuesta, no verifica hechos.
   T-027 anade verificacion de items para cerrar esta brecha.

## Decisiones tomadas

- Arquitectura: `create_retrieval_chain` (LangChain) ‚Äî deterministico, incondicional
- Memoria: `RunnableWithMessageHistory` (LangChain) ‚Äî sin SessionManager custom
- Multi-idioma: delegado 100% al LLM via system prompt ‚Äî sin codigo
- Vector DB: ChromaDB SQLite ‚Äî sin Qdrant/Milvus
- Ingesta: script manual `update_knowledge.sh` ‚Äî sin watchdog
- Modelo actual: `llama-8b` ‚Üí propuesto `llama-3.3-70b-versatile` (T-027)
- Evaluador: RAGAS triada (Context Relevance, Faithfulness, Answer Relevance)
